# 通用邮件通知与邮箱验证码设计

## 1. 目标与范围

本次仅实现后端能力，不修改前端。目标是将现有违章 SMTP 通知扩展为通用邮件能力，并提供安全的注册邮箱验证和找回密码流程。

本次包含：

- 通用模板邮件发送、SMTP 安全连接和发送日志。
- 保留并迁移现有违章通知到通用邮件服务。
- 注册邮箱验证码发送与校验。
- 找回密码验证码发送与密码重置。
- 邮箱全局唯一、注册必填邮箱。
- 重置密码后使旧登录令牌失效。

本次不包含：

- 前端页面或交互修改。
- 注册成功欢迎邮件。
- 短信通知。
- Celery 异步发送和自动重试队列。

## 2. 核心规则

- 验证码为 6 位数字，有效期 10 分钟。
- 同一邮箱、同一用途 60 秒内不得重复发送。
- 同一邮箱、同一用途仅最新一枚成功发送的验证码有效。
- 单枚验证码最多允许 5 次失败尝试。
- 验证码仅保存带随机盐的安全哈希，不保存明文。
- 验证码按用途隔离：`register` 与 `password_reset` 不可混用。
- 验证码成功消费后不可再次使用。
- 用户邮箱必填且全局唯一，比较前去除首尾空格并转为小写。
- 不在通知日志、应用日志或 API 响应中暴露验证码、SMTP 密码等秘密。

## 3. 架构

### 3.1 SMTP Provider

保留 `NotificationProvider` 抽象，并完善 `EmailSmtpProvider`：

- 使用 Python 标准库 `email.message.EmailMessage` 构造 MIME 邮件，保证中文主题和正文正确编码。
- 支持 `starttls`、`ssl`、`none` 三种连接安全模式。
- 支持连接超时。
- 仅从环境变量读取 SMTP 主机、端口、用户名、密码、发件地址和安全模式。
- 将可预期的连接、认证和发送异常转换为 `SendResult(status="failed", error=...)`，不向业务层泄漏异常。

### 3.2 通用邮件服务

将现有 `NotificationService` 泛化，提供按模板代码发送邮件的入口。服务负责：

1. 读取启用的邮件模板。
2. 使用受控上下文渲染主题和正文。
3. 调用 `NotificationProvider`。
4. 写入发送结果日志。

违章通知继续通过 `violation_notice` 模板发送。新增模板：

- `register_email_code`
- `password_reset_email_code`

认证邮件的通知日志仅记录模板代码和非敏感摘要，不保存渲染后包含验证码的正文。模板缺失、禁用或变量错误应产生明确的失败结果和安全的内部错误信息。

### 3.3 邮箱验证码服务

新增独立的验证码服务，负责生成、发送、校验和消费验证码。该服务依赖数据库会话与通用邮件服务，不直接依赖 SMTP 实现。

发送流程：

1. 规范化邮箱并校验用途。
2. 检查 60 秒冷却时间。
3. 生成 6 位验证码及安全哈希。
4. 调用通用邮件服务发送明文验证码。
5. 仅在发送成功后持久化新验证码并使同用途旧验证码失效。

校验与消费流程在同一数据库事务内完成：锁定最新有效记录，检查有效期、使用状态和尝试次数，验证哈希；失败时递增尝试次数，成功时设置使用时间。注册用户或更新密码与验证码消费必须原子提交。

## 4. 数据模型

### 4.1 `email_verification_codes`

新增字段：

- `id`: 主键。
- `email`: 已规范化的邮箱，建立查询索引。
- `purpose`: `register` 或 `password_reset`。
- `code_hash`: 验证码安全哈希。
- `attempt_count`: 失败次数，默认 0。
- `expires_at`: 过期时间。
- `used_at`: 成功使用或被新验证码替代的时间，可空。
- `created_at`: 创建时间。

查询索引覆盖 `(email, purpose, created_at)`。业务层锁定最新有效记录，数据库约束限制用途取值。

### 4.2 `users`

- `email` 改为非空。
- 对规范化后的 `email` 增加唯一约束。
- 新增 `auth_version` 整数字段，默认 0。

迁移执行前检查已有空邮箱和重复邮箱。若存在问题，迁移明确中止并输出清理提示，不自动修改、合并或删除用户。

### 4.3 `notifications`

- `violation_id` 改为可空，以支持注册和找回密码邮件。
- 新增 `template_code`，标识邮件类型。
- `owner_id` 保持可空，注册验证码发送时用户尚不存在。
- `content` 对认证邮件保存非敏感摘要；违章邮件保持现有可审计内容。

## 5. API 合约

### 5.1 发送注册验证码

`POST /api/v1/auth/register/email-code`

请求：

```json
{ "email": "user@example.com" }
```

行为：邮箱已注册返回 `409`；冷却期内返回 `429`；SMTP 或模板发送失败返回 `503`；成功返回 `202`。

### 5.2 注册

`POST /api/v1/auth/register`

请求在原字段基础上要求：

```json
{
  "username": "newuser",
  "password": "pass1234",
  "email": "user@example.com",
  "verification_code": "123456"
}
```

邮箱改为必填。验证码无效、过期、已使用或尝试次数耗尽统一返回 `400` 和模糊错误信息。用户名或邮箱冲突返回 `409`。成功时创建用户与消费验证码原子提交，并保持现有注册成功响应。

### 5.3 发送找回密码验证码

`POST /api/v1/auth/password-reset/email-code`

请求：

```json
{ "email": "user@example.com" }
```

无论邮箱不存在、账号禁用、处于冷却期或内部邮件发送失败，外部均返回相同的 `202` 模糊响应，避免根据状态、内容或错误码枚举账号。实际发送失败写入通知日志，供运维排查。

### 5.4 重置密码

`POST /api/v1/auth/password-reset`

请求：

```json
{
  "email": "user@example.com",
  "verification_code": "123456",
  "new_password": "newpass5678"
}
```

验证码相关失败统一返回 `400`。成功时更新密码哈希、消费验证码并递增 `auth_version`，三者原子提交。

## 6. 登录令牌失效

登录签发的 JWT 增加 `auth_version` 声明。认证依赖在解析 JWT 后，将声明值与当前用户的 `auth_version` 比较；不一致返回 `401`。找回密码重置成功后递增版本，之前签发的 JWT 立即失效。正常登录、个人中心修改密码和管理员重置密码也应递增版本，保持安全语义一致。

## 7. 错误与事务边界

- 注册验证码发送失败时不写入新验证码，也不使旧验证码失效。
- 找回密码对外采用固定响应；内部发送结果仍完整记录。
- 验证码失败计数必须提交，不能因随后抛出的业务异常回滚。
- 注册和密码重置使用行锁与数据库唯一约束防止并发重复消费。
- 邮件发送不放在持有用户行锁的长事务中；验证码消费才进入短事务。
- SMTP 错误对客户端返回稳定的业务错误，不返回服务器地址、认证详情或原始堆栈。

## 8. 配置

新增或完善以下环境变量：

- `SMTP_HOST`
- `SMTP_PORT`
- `SMTP_USER`
- `SMTP_PASSWORD`
- `SMTP_FROM`
- `SMTP_SECURITY`: `starttls`、`ssl` 或 `none`
- `SMTP_TIMEOUT_SECONDS`
- `EMAIL_CODE_TTL_SECONDS`，默认 600
- `EMAIL_CODE_RESEND_SECONDS`，默认 60
- `EMAIL_CODE_MAX_ATTEMPTS`，默认 5

无 SMTP 配置时 provider 返回 `smtp_not_configured`，应用仍可启动。

## 9. 测试策略

### Provider 单元测试

- 中文主题和正文使用标准 MIME 编码。
- `starttls`、SSL 和无加密模式选择正确。
- 需要认证时执行登录，无认证配置时不登录。
- 缺配置、连接失败、认证失败和发送失败均返回稳定失败结果。

### 验证码服务测试

- 生成 6 位验证码且数据库不保存明文。
- 10 分钟过期、60 秒冷却、最新验证码生效。
- 用途隔离、最多 5 次尝试、一次性消费。
- 新邮件发送失败时旧验证码仍有效。
- 并发或重复消费只有一次成功。

### API 与认证测试

- 注册必须提供邮箱和正确验证码。
- 邮箱规范化及唯一约束。
- 找回密码请求不泄露账号存在、禁用、冷却或发送失败状态。
- 密码重置后新密码可登录，旧密码不可登录，旧 JWT 返回 `401`。
- 已登录修改密码和管理员重置密码同样使旧 JWT 失效。
- 现有违章通知发送与记录行为保持通过。

### 迁移测试

- 全新数据库可升级到最新版本。
- 合法旧数据可迁移。
- 空邮箱或重复邮箱产生明确的迁移失败提示。

## 10. 验收标准

- 三个新接口和增强后的注册接口符合上述合约。
- 数据库中无验证码明文，日志中无敏感认证邮件正文。
- 验证码有效期、冷却、尝试限制、用途隔离与一次性消费均由自动化测试证明。
- 邮箱全局唯一且注册必填。
- 密码重置后所有旧 JWT 失效。
- 配置真实 SMTP 后可发送中文注册、重置和违章邮件。
- 后端完整测试套件通过，且不修改前端代码。
