## 1. 背景与目标

项目为绿地新建（greenfield），当前仓库仅含 `README.md`、`docs/`、空 `.agents/`，无任何代码。

完整系统设计见 `docs/系统设计方案/`，覆盖前端、后端 API、AI 服务、异步任务、视频流管线、统计、系统管理等全部子系统。完整系统体量大，按 superpowers 做法**先拆分、再针对第一个子项目做详细方案**。

本方案是**第一阶段**：聚焦后端 + AI 主干流水线，跑通"图片接入 → AI 识别 → 规则判定 → LLM 初审 → 人工终审 → 归档 → 通知"的最小闭环，前端只做联调所需的最小页面。视频流管线、多模态复核、完整统计/系统管理页推迟到后续阶段。

设计文档中的核心正确性约束在本阶段严格遵守：**YOLOv8 输出的 confidence 是目标检测置信度，不是违章置信度**；违章成立必须由规则判定服务形成证据链 + LLM 初审意见 + 人工终审共同支撑，不得用 YOLOv8 置信度直接判定违章。

## 2. 第一阶段范围边界

### IN（本阶段做）

- FastAPI 应用骨架 + 配置管理 + JWT/RBAC 鉴权
- DB schema + Alembic 迁移（第一阶段所需核心表 + 辅助表子集，见 §5）
- `/api/v1/*` 业务路由：auth、intakes（市民/摄像头/后台上传）、cases（列表/详情/通过/驳回/重审）、violations（列表/详情/车主查询）、statistics（基础统计接口，无可视化）
- `/internal/ai/*` AI 路由 + 可插拔适配器：YOLOv8 预训练、PaddleOCR、真实 LLM（智谱 GLM）、规则判定
- 违章规则判定：超速 + 占用专用车道
- Celery 异步任务流：detect → ocr → evaluate → review → send_notification
- 案件状态机 + 审计日志
- 真实邮件通知（SMTP，替代短信）
- 最小前端联调页：登录 + 上传 + 案件卡片 + 审核通过/驳回

### OUT（推迟到后续阶段）

- 多模态复核 `/internal/ai/review/vision`
- 闯红灯 / 压线 / 逆行 / 违停规则（需红绿灯/车道线检测、自训练模型或单图无法证明时长）
- 20fps 视频流抽帧 / ROI / 跟踪 / 候选事件触发管线（`trigger_candidate_event_task`）
- 真实短信服务商接入（个人开发者无法开通云短信，永久不可行；用邮件替代）
- LLM 分析报告（`generate_report_task`）作为可选 stretch goal
- 完整统计可视化台、系统管理页
- 摄像头 API Key + 签名 + 时间戳防重放（本阶段用简化 API Key 鉴权）
- 生产部署/Docker（本地跑通为主，docker-compose 作为可选交付物）

> 注：超速违章依赖摄像头上传的 `speed` 元数据，因此摄像头单图抓拍接入（`/intakes/camera-captures`）在第一阶段范围内。

| 前端 | Vue3 + Element Plus |

| 测试 | 核心逻辑 TDD，非核心 test-after |

| 车牌检测 | 预训练车牌检测模型框车牌 + PaddleOCR 识别字符 |

### 3.1 关键约束与说明

- **真实短信不可行**：个人开发者无法开通阿里云/腾讯云短信（需企业资质）。通知模块改用邮件 SMTP 真实发送，`NotificationProvider` 抽象保留，未来有企业资质可再加 `SmsProvider`。
- **违停在单图下不可判定**：单张图片无法证明"连续静止时长"，故第一阶段不做违停，`violation_rules` 本阶段只需 `speed` / `special_lane` 两种 rule_type。
- **预训练 YOLOv8 的局限**：预训练 COCO 模型能检车辆/行人/车牌框，但检不准红绿灯状态、停止线、车道线。因此闯红灯/压线等需这些要素的违章类型推迟到能自训练交通专用模型之后。本阶段的两种违章类型不依赖这些要素：超速靠速度元数据，占用专用车道靠车辆框 + 配置 ROI。

## 4. 整体架构与目录结构

单体 FastAPI 应用：业务路由 + AI 路由同进程，Celery worker 独立进程，共享同一套 `app/` 代码与 Redis broker。AI 逻辑在 `app/ai/` 内，worker 直接 import 调用，无跨服务 HTTP。

```text
backend/
├── app/
│   ├── api/              # /api/v1/* 业务路由
│   │   └── v1/{auth,intakes,cases,violations,statistics}.py
│   ├── ai/               # /internal/ai/* AI 路由 + 适配器
│   │   ├── routes.py     # /internal/ai/yolo|ocr|rules|review/*
│   │   ├── adapters/     # 可插拔接口 + 实现
│   │   │   ├── base.py   # YoloDetector/OcrEngine/RuleEvaluator/LLMProvider/NotificationProvider 抽象
│   │   │   ├── yolo_ultralytics.py
│   │   │   ├── ocr_paddle.py
│   │   │   ├── llm_provider.py   # OpenAI 兼容（智谱 GLM，env 配置）
│   │   │   └── email_smtp.py
│   │   └── rules/        # 违章规则判定（超速/占用专用车道）
│   ├── tasks/            # Celery 任务（detect/ocr/evaluate/review/send_notification）
│   ├── models/           # SQLAlchemy 模型
│   ├── schemas/          # Pydantic 请求/响应
│   ├── core/             # config/security(JWT,RBAC)/deps/audit
│   ├── worker.py         # Celery app 实例
│   └── main.py           # FastAPI app
├── alembic/              # DB 迁移
├── tests/                # 单元 + 集成（TDD）
├── pyproject.toml
└── .env.example
frontend-min/             # 最小联调页（Vue3 + Element Plus + Vite）
```

进程模型：`uvicorn app.main:app`（API）+ `celery -A app.worker worker`（任务）+ 本地 Redis + 本地 MySQL。

### 5.4 `violation_rules` 表结构（设计文档未细化，本方案补充）

```text
violation_rules
├── id
├── rule_code          # SPD-001 / LANE-001
├── violation_type     # 超速 / 占用专用车道
├── rule_type          # speed / special_lane
```

## 7. AI 可插拔适配器

每个 AI 组件抽象成接口（`app/ai/adapters/base.py`），第一阶段实现一套，后期可换：

```python
class YoloDetector(ABC):
    def detect(self, image_path) -> DetectionResult  # objects[], annotated_image_url

class OcrEngine(ABC):
    def recognize_plate(self, plate_crop_path) -> str | None

class RuleEvaluator(ABC):
    def evaluate(self, detection, ocr, intake_event, rule) -> RuleResult

class LLMProvider(ABC):           # 一套 OpenAI 兼容协议，env 切 provider
    def review(self, evidence_payload) -> AIReviewResult  # conclusion/reason/confidence

class NotificationProvider(ABC):
    def send(self, to_email, content) -> SendResult
```

第一阶段实现：

| 接口 | 实现 | 说明 |
| --- | --- | --- |
| YoloDetector | `UltralyticsYoloDetector` | yolov8n 预训练 COCO 检车辆/人 + 独立预训练车牌检测模型框车牌 |
| OcrEngine | `PaddleOcrEngine` | 裁剪车牌框 → PaddleOCR 识别字符 |
| RuleEvaluator | `RoiRuleEvaluator` | 超速 + 占用专用车道（见 §8） |
| LLMProvider | `OpenAICompatibleLLMProvider` | openai SDK，base_url=智谱，model=glm-4-flash，env 配 key |
| NotificationProvider | `EmailSmtpProvider`（真实）+ `LogNotificationProvider`（兜底） | SMTP env 配置 |

## 8. ROI 规则判定逻辑

### 8.1 超速（不依赖 YOLO 视觉要素，靠元数据）

```text
输入: intake_event.speed, violation_rule(speed_limit)
判定: speed > speed_limit → rule_matched=True
证据: ["检测到车辆"(YOLO), f"车速{speed}超过限速{limit}"]
完整度: complete（有车 + 有速度 + 超阈值）
```

### 8.2 占用专用车道（YOLO + ROI 几何）

```text
输入: detection.vehicle_bbox, violation_rule(lane_roi, allowed_vehicle_types, time_window)
判定: vehicle_bbox 中心点落在 lane_roi 内
      AND 当前时间在 time_window 限行时段
      AND 车辆类型不在 allowed_vehicle_types → rule_matched=True
证据: ["检测到车辆","车辆位于专用车道ROI内","当前时间属限行时段"]
完整度: complete
```

> 车辆类型取自 YOLOv8 类别标签（car/truck/bus/motorcycle 等 COCO 类别）。例如占用公交车道：`allowed_vehicle_types=["bus"]`，YOLO 判为 `car` 则构成违章。

### 8.3 证据不完整的处理

证据不完整（如 YOLO 未检到车辆）→ `evidence_level=insufficient`，规则不匹配，案件走"疑似但证据不清"路径。第一阶段不做多模态复核，所以这类案件直接标 `need_review` 进人工队列，由人工判断。

## 9. Celery 任务流与案件状态机

### 9.1 任务流

```text
intake 接口（同步）
   └─ chain(
        detect_objects_task,          # YOLO 检测 → ai_detection_results
        ocr_plate_task,               # OCR 车牌（检到车牌框才跑）
        evaluate_violation_rule_task, # 规则判定 → violation_rule_results
        ai_review_task,               # LLM 初审 → ai_review_results，状态→pending_human_review
      ).on_fail(标记案件 AI 初审失败，进人工队列)

人工 approve（同步接口）
   └─ send_notification_task          # 邮件 SMTP → notifications
```

- 用 Celery `chain` 编排四步依赖（detect 的输出是 ocr 的输入）。
- 任一步失败：案件状态置 `pending_human_review` + 卡片标"AI 初审失败，需人工判断"（对齐设计文档失败处理）。
- `send_notification_task` 由人工通过接口触发，不在主链上。

### 9.2 案件状态机

```text
uploaded ──(投递detect任务)──> detecting ──(detect完成)──> ai_reviewing
   │                                                              │
   │                                              (ocr+evaluate+llm完成)
   │                                                              ▼
   │                                                    pending_human_review
   │                                                          │        │
   │                                                  approve  │        │ reject
   │                                                          ▼        ▼
   │                                                       archived  rejected
   │                                                          │
   │                                                  (send_notification)
   │                                                          ▼
   └─────────────────────────────────────────────────────── notified
```

- `detecting`/`ai_reviewing` 期间任务失败 → 回落 `pending_human_review`（带失败标记）。
- `request-recheck` 接口从 `pending_human_review` 重新投递 detect 链。

## 10. 鉴权与安全

| 项 | 第一阶段做法 |
| --- | --- |
| 登录鉴权 | JWT access token（env 配过期时间），`/auth/login`、`/auth/me` |
| RBAC | 四角色 citizen/reviewer/admin/camera；依赖注入校验角色；`/permissions/menus` 用静态角色→菜单映射 |
| 摄像头接入 | 简化 API Key：`X-Camera-Key` header → 查 `camera_api_keys` → 设备。签名+时间戳防重放推迟 |
| 图片校验 | jpg/png/webp、大小限制（10MB）、MIME magic number 校验 |
| 审计日志 | approve/reject/intake/login 等敏感操作写 `audit_logs` |
| AI 结果不可变 | `ai_detection_results`/`violation_rule_results`/`ai_review_results` append-only，人工只追加 `cases` 审核字段，不覆盖 AI 表 |
| 通知模板 | `notification_templates` 渲染内容（模板 + 违章数据），不允许任意文本发送 |

权限原则（来自上游设计，本阶段遵循）：

- 市民只能访问本人数据。
- 摄像头只能调用接入接口，不能查询业务数据。
- 工作人员可以审核案件，但不能修改 AI 原始识别结果，只能追加人工结论。
- 正式违章必须由人工终审通过后生成。

## 12. 最小前端联调页（`frontend-min/`）

Vue3 + Element Plus + Vite + Pinia + Vue Router + Axios。只做能跑通全链路的几页：

- 登录页
- 上传页（citizen / camera / admin 三种上传）
- 案件列表 + 详情卡片：原图 / 标注图 / YOLO 结果 / OCR 车牌 / 规则判定 / LLM 初审意见

## 15. 第一阶段验收标准

- 市民上传和摄像头单图抓拍均能进入统一案件流水线。
- 摄像头 20fps 视频流管线**不**在本阶段范围（推迟），单图接入闭环完成即可。
- YOLOv8 只输出目标检测结果，系统不使用目标检测置信度直接判断违章。
- 超速（速度元数据超阈值）和占用专用车道（车辆框落在专用车道 ROI + 限行时段）能形成完整证据链。
- 规则匹配且证据完整的案件能生成 LLM 初审意见（智谱 GLM）并进入人工终审。
- 证据不完整的案件标 `need_review` 进人工队列（本阶段不调多模态复核）。
- 审核工作台（最小前端）能轮询刷新显示案件卡片。
- 人工通过后生成正式违章记录，AI 原始结果不可被覆盖。
- 归档后能调用真实邮件 SMTP 接口并记录通知结果（`pending/sent/failed`）。
- 工作人员能查看违章列表和基础统计数据。
- 鉴权、RBAC、图片校验、审计日志均生效。
