# 端到端浏览器完整测试计划

> **测试目标:** 用 Playwright MCP 逐条验证全部功能的浏览器端行为
> **测试环境:** 后端 localhost:8000 / 前端 localhost:5174 / MySQL localhost:3306
> **测试数据:** seed_data.py 提供的演示数据 (admin/admin1234 + reviewer + citizens)
> **覆盖范围:** 认证、RBAC 三大角色、市民举报、审核流程、管理后台、AI stub 接口

---

## 测试 Group 1: 登录 & 注册

### 1.1 登录成功
- **步骤:** 访问 `http://localhost:5174/login` → 输入 admin/admin1234 → 点击登录
- **预期:** 跳转 `/admin/stats`，左侧菜单全量显示，右上角显示"超级管理员"
- **验证点:** URL 路径、菜单数量、角色标签

### 1.2 错误密码
- **步骤:** 输入 admin/wrongpassword → 点击登录
- **预期:** 弹错误提示"用户名或密码错误"，停留在 /login

### 1.3 未登录访问保护页
- **步骤:** 清空 localStorage → 直接访问 `/admin/stats`
- **预期:** 重定向到 `/login`

### 1.4 注册新市民账号
- **步骤:** 点"立即注册" → 输入新用户名 test_citizen_01 + 密码 → 注册
- **预期:** 成功跳转登录页

## 测试 Group 2: RBAC 角色权限

### 2.1 Admin 菜单全量

- **验证:** 仅有"随手拍举报"、"我的举报"、"我的违章"、"车辆绑定"

## 测试 Group 4: 审核员审核流程

### 4.1 审核工作台
- **步骤:** 以 admin 或 reviewer 登录 → 进入"案件审核"
- **验证:** 列表展示案例卡片，显示 AI 审查状态标签(建议通过/需复核/建议驳回)

### 4.2 案件详情 & AI 管道展示
- **步骤:** 点击一个案例
- **验证:** 左侧展示原图+AI标注图，右侧展示 YOLO检测→OCR→规则判定→AI初审 四步管道

### 4.3 审核通过
- **步骤:** 输入车牌号、违章类型、罚款金额、扣分 → 点击通过
- **预期:** Toast 提示成功，案例状态变为"已通过"

### 4.4 审核驳回
- **步骤:** 选一个 pending 案例 → 输入驳回原因 → 点驳回
- **预期:** 案例状态变为"已驳回"

### 4.5 请求 AI 复核
- **步骤:** 点"请求 AI 复核"
- **预期:** 提示"AI 复核已排队"（当前 stub 模式下的预期行为）

## 测试 Group 6: AI 内部接口 (stub 模式)

### 6.1 YOLO 检测
- **步骤:** 用 curl 调 `POST /api/v1/internal/ai/yolo/detect` (带 JWT token + 图片)
- **预期:** 返回 stub 数据：objects=[{label:"car", confidence:0.92}], vehicle_bbox=[100,200,300,350], plate_bbox=[120,230,200,270]

### 6.2 OCR 车牌识别
- **步骤:** 调 `POST /api/v1/internal/ai/ocr/plate`
- **预期:** 返回 {plate_no: "京A12345"}

### 6.3 规则评估
- **步骤:** 调 `POST /api/v1/internal/ai/rules/evaluate`
- **预期:** 返回 rule_matched=true, evidence_level="complete"

### 6.4 LLM 审查
- **步骤:** 调 `POST /api/v1/internal/ai/review/text`
- **预期:** 返回 conclusion="suggest_approve", ai_confidence=0.88
